# Hive MITM Proxy Cert Issue

## Symptoms

In `kubectl logs -n hive deploy/hive`:

```
proxy client TLS handshake failed: local error: tls: bad record MAC
proxy client TLS handshake failed: remote error: tls: bad certificate
[entrypoint] WARN: proxy CA install via gosu failed
[entrypoint] WARN: iptables chain creation failed (need NET_ADMIN capability)
```

Agents that push to GitHub via `git` (which routes through the proxy) get TLS
errors. PR creation and git push may fail.

## Root cause

Hive uses a MITM HTTPS proxy on `127.0.0.1:18443` to intercept and log all
outbound HTTPS traffic from agents. The proxy generates a self-signed CA
certificate (`/data/proxy-ca.pem`) on first run and forges per-host certs on
the fly. For agents to trust these forged certs, the CA cert must be in the
system trust store.

The chain of failures:

### 1. iptables transparent proxy blocked

The entrypoint tries to set up iptables `REDIRECT` rules to force all
outbound `:443` traffic through the proxy:

```bash
iptables -t nat -N HIVE_PROXY
iptables -t nat -A HIVE_PROXY -p tcp --dport 443 -j REDIRECT --to-ports 18443
```

This fails with:

```
[entrypoint] WARN: iptables chain creation failed (need NET_ADMIN capability)
```

The container runs under a PodSecurity `restricted` policy that drops
`NET_ADMIN` (and all other non-essential capabilities). Without iptables,
traffic only routes through the proxy if the client explicitly sets
`HTTPS_PROXY` / `http.proxy`.

### 2. Git proxy works but cert not trusted

Git IS configured to use the proxy:

```ini
[http]
    sslCAInfo = /data/proxy-ca.pem
    proxy = http://127.0.0.1:18443
[https]
    proxy = http://127.0.0.1:18443
```

So git sends `CONNECT` requests through the proxy. But the proxy's HTTPS
listener expects TLS on the wire, not plain-text HTTP CONNECT — causing
`tls: bad record MAC` when the proxy tries to parse the CONNECT as a TLS
ClientHello.

### 3. CA cert install fails as non-root

The entrypoint drops from root to the `dev` user early, then later tries to
install the CA cert into the system trust store:

```bash
gosu root sh -c 'cp /data/proxy-ca.pem /usr/local/share/ca-certificates/... && update-ca-certificates'
```

This fails because the container has **no effective capabilities**:

```
CapEff: 0000000000000000
CapBnd: 00000000a80425fb
```

`CAP_SETUID` is not in the bounding set, so `su-exec` (the underlying binary
behind `gosu`) cannot switch to root from a non-root user. The fallback
`cp` also fails since `/usr/local/share/ca-certificates/` is root-owned.

### 4. SSL_CERT_FILE only set in .bashrc

The entrypoint sets `SSL_CERT_FILE=/data/proxy-ca.pem` in
`/data/home/.bashrc`, but this only takes effect for interactive login
shells. Processes spawned directly (Go binary, git child processes started
by pi) don't source `.bashrc`, so they don't get the env var.

## Current workaround

The CA cert exists at `/data/proxy-ca.pem` and is valid (self-signed,
expires June 2027). The gitconfig has `sslCAInfo` pointing to it. For
processes that use Go's `x509.SystemCertPool()` or `SSL_CERT_FILE`, the
cert is not trusted.

## Fix

Two changes needed:

### Fix 1: Install CA while still root

In [`v2/deploy/entrypoint.sh`] (the upstream fork), move the CA install step
from the non-root section (currently at line ~434, after the `gosu dev` drop)
to the root section, **before** the `exec gosu dev` line. The proxy CA cert
is first generated by the Go binary at `/data/proxy-ca.pem` and persisted
across restarts (it was generated in a previous run, so it'll be present on
restart). For brand-new deployments, the Go binary will generate it on first
startup.

Concretely, insert right before the `exec gosu dev` line (~line 401):

```bash
# Install the MITM proxy CA while still root (gosu root won't work later
# because the container lacks CAP_SETUID).
if [ -f /data/proxy-ca.pem ]; then
  cp /data/proxy-ca.pem /usr/local/share/ca-certificates/hive-proxy-ca.crt 2>/dev/null && \
    update-ca-certificates 2>/dev/null && \
    echo "[entrypoint] proxy CA installed to system trust store" || \
    echo "[entrypoint] WARN: proxy CA install failed"
else
  echo "[entrypoint] WARN: proxy CA not found at /data/proxy-ca.pem (will be generated by Go binary)"
fi
```

Also remove the duplicate `gosu root`-based CA install from the non-root
section (line ~434).

### Fix 2: Set SSL_CERT_FILE globally

After the `exec gosu dev` drop, export `SSL_CERT_FILE` directly (not via
`.bashrc`) so all child processes inherit it:

```bash
export SSL_CERT_FILE=/data/proxy-ca.pem
export NODE_EXTRA_CA_CERTS=/data/proxy-ca.pem
```

### Fix 3 (optional): Skip iptables on restart

The `iptables` attempt fails noisily every startup. Either add `NET_ADMIN`
to the container's `securityContext.capabilities` in the deployment manifest,
or suppress the warning since it's expected in this environment.
